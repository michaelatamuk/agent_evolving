Layer 1 — Episodic Accumulation  Reacts to failures by enriching runtime context — S is never rewritten
MUSE-Autoskill
Self-Evolving Agents via Skill Creation, Memory, Management & Evaluation
Lin, H. et al. · 2026
arXiv:2605.27366 ↗Layer 1
Most complete episodic management framework. Tracks experiences across runs, enables cross-agent record transfer, gates direct edits with unit tests and runtime feedback. The strongest expression of what the episodic paradigm can become — and the clearest demonstration of what it cannot: S is never rewritten. Unit tests on proposed edits + runtime feedback. No standardized external benchmark. Before vs. after comparison on held-out instances. Test gate is scalar. SAGE does what MUSE cannot: rewrites S via evolutionary macro search. Complementary — MUSE handles reactive Layer 1 record capture; SAGE handles periodic structural S rewriting.
Layer 2 — Localized Semantic Editing  Writes corrections into S from localized failure signals — no holistic multi-dimensional regression gate
SkillOpt
Executive Strategy for Self-Evolving Agent Skills
Zhu et al. · 2026
arXiv:2605.23904 ↗Layer 2
Analyzes failure trajectories to find the contrastive failure point — the moment output diverges from expected. Proposes a minimal targeted edit to S, committed only if it clears a held-out split. Surgical: smallest fix for the identified defect. Held-out split with scalar quality score. Task success rate before vs. after edit. No external benchmark; evaluated on own skill domain. SAGE uses macro population search vs. single-point patching. SAGE’s rubrics holdout with per-dimension no-regression gate catches multi-dimensional regressions that SkillOpt’s scalar gate silently accepts.
SkillEvolver
Skill Learning as a Meta-Skill
Zhang, G. et al. · 2026
arXiv:2605.10500 ↗Layer 2
Generates K=4 diversified skill variants, collects labeled pass/fail traces, applies contrastive patches from comparing success vs. failure traces. A 9-check auditor validates each patch — most explicit verification gate in the local editing class. 9-check binary auditor per patch + labeled pass/fail execution traces. K=4 variants compared against each other and vs. baseline. SAGE’s evolutionary population mirrors K-variant diversification at macro structural scope. Replaces 9 binary checks with a 7-metric continuous fitness family + per-dimension no-regression constraint.
Layer 3 — Generative Macro Synthesis  Synthesizes new skill structures via evolutionary or causal search — calibrated on scalar benchmarks, not production quality dimensions
EvoSkill
Automated Skill Discovery for Multi-Agent Systems
Alzubi, S. et al. · 2026
arXiv:2603.02766 ↗Layer 3
Three collaborating agents (A1 analyzes failures, A2 synthesizes structures, A3 validates) create entirely new skill folders — not patching existing text but discovering new skill structures from scratch. Only top-K admitted by a held-out frontier scalar score. Held-out validation frontier; top-K selection by aggregate scalar score. Three-agent system evaluated collectively; no per-dimension breakdown. Both use multi-agent macro synthesis. SAGE adds: 7-metric fitness (vs. scalar), per-dimension no-regression gate (vs. top-K scalar), Thompson Sampling adaptive budget allocation.
SkillGen
Verified Inference-Time Agent Skill Synthesis
Ma, Y. et al. · 2026
arXiv:2605.10999 ↗Layer 3
Frames synthesis as causal intervention: “what skill text causes these repairs without causing those regressions?” Tracks failures fixed AND successes broken on identical held-out instances. Generation-verification-refinement loop. Most causally rigorous objective in the class. Held-out benchmark tracking repair count AND regression count simultaneously on same instances. Most explicit regression-awareness among all 6 papers. Closest predecessor to SAGE — both track what an improvement breaks. SAGE extends: 7 continuous per-axis metrics (vs. binary counts), Bayesian acceptance gate, Thompson Sampling adaptive budget.
SkillMAS
Skill Co-Evolution with LLM-based Multi-Agent System
Pan, S. et al. · 2026
arXiv:2605.09341 ↗Layer 3
Only framework to argue skill content and multi-agent routing topology are structurally coupled and must be co-evolved together. Simultaneously optimizes skill text, routing, agent roles, and pipeline organization against verified execution traces. Verified execution traces + benchmark metrics for the co-evolved system as a whole. Attribution between skill text vs. routing improvement not reported separately. SAGE optimizes S only. SkillMAS’s co-evolution insight (S + routing topology) is an open future direction. SAGE’s multi-dimensional fitness apparatus could be applied to SkillMAS-style co-evolution.
Synthesis — What These 6 Papers Tell Us Together
  • The direction is settled. Every paper accepts that agents must improve their own skills. The disagreement is purely architectural.
  • Each layer solves the last layer’s problem. MUSE-Autoskill proves episodic management has a ceiling → SkillOpt and SkillEvolver write into S → EvoSkill/SkillGen/SkillMAS escape local-edit constraints with macro search. Each transition is architecturally motivated.
  • Evaluation is the sticking point. Every Layer 2 and Layer 3 paper uses a scalar held-out gate or binary pass/fail. None independently measures format, entity retention, ordering, and semantic equivalence. This is exactly the gap SAGE closes.
  • SkillGen is the closest predecessor. Only paper to explicitly track regression alongside improvement — the same instinct as SAGE’s no-regression constraint, but binary rather than continuous per quality dimension.
  • SkillMAS opens the next question SAGE does not yet answer: if routing topology and skill text are coupled, optimizing S alone may miss system-level improvements. An open research direction.

Goodhart's Law Collapse Simulator
Why gaming a quality test is dangerous — and how to catch it
The core idea

Imagine you manage a call center and decide to measure employee performance by how many calls they close per hour. Simple enough. But employees quickly learn they can boost their number by hanging up on difficult customers instead of helping them. Their score goes up — but the actual quality of service collapses.

This is called Goodhart's Law: "When a measure becomes a target, it ceases to be a good measure." The same thing happens with AI systems that improve themselves using automatic quality tests — if the test can be gamed, the AI will game it.

This simulator shows that exact scenario playing out in slow motion, so you can see the danger with your own eyes.

What you see on the chart
Teal solid line — Score of a skill trained on a reliable test that reflects genuine quality
Teal dashed line — Actual real-world quality of the same skill (tracks the score — no gap)
Orange solid line — Score of a skill trained on a shortcut test. Rises fast — looks impressive
Red dashed line — Actual real-world quality of the shortcut skill. Hidden until you reveal it
X →
Each step = one round of training (called a "generation")
Y ↑
Score / quality level (higher = better, max = 1.0)
What to try — step by step
1
Click "Show real quality (orange)" in the right panel. This arms the reveal — the hidden red line will appear from generation 1.
2
Now click "Run Simulation". Watch all four lines rise together at first. The orange solid line (shortcut score) rises faster than teal.
3
Around generation 13, watch the red dashed line (orange skill's real quality) start to fall — even while the orange solid score keeps climbing. This is the collapse.
4
By generation 25: the orange skill scores ~90% but has ~10% real quality. The teal skill scores ~85% and has ~80% real quality. Shortcut "won" the test. It lost at the actual job.
5
Try running without revealing: the shortcut looks like the clear winner. That's what you'd see if you only looked at test scores — a dangerous blind spot.
Why this matters for SAGE

SAGE is an AI system that improves itself automatically. Every improvement cycle runs thousands of mutations and picks the "best" ones by test score. If the test is gameable, the system will silently degrade over time while appearing to improve.

SAGE prevents this with a multi-dimensional holdout gate: each candidate must pass 7 separate quality checks on data it has never seen before, including checks for accuracy, format, entity retention, and ordering. Gaming all 7 simultaneously is practically impossible.

Gate Composition Calculator
How stacking independent quality checks multiplies deployment safety
The core idea

Before SAGE deploys an improved skill to users, it must be confident the new version is actually better — not just superficially different. It runs every candidate through a series of quality checks called "gates."

Think of airport security. One metal detector catches most threats but occasionally misses some. Two independent detectors — because their weaknesses are different — together catch almost everything. The key word is independent: when tests fail for different reasons, combining them multiplies their effectiveness.

This calculator lets you set up three independent tests and see exactly how safe the combined system is.

What you see on the chart

The chart has two axes. You want to be in the bottom-left corner — that means both risks are low:

X →
Horizontal axis (False Accept Rate) — % of bad skill upgrades that slip through all tests and get deployed by mistake. Lower = safer.
Y ↑
Vertical axis (False Reject Rate) — % of good skill upgrades that get incorrectly blocked. Lower = less waste.
Teal dot (S1) — risk level using only 1 quality test
Blue dot (S1∧S2) — risk level using 2 tests combined (must pass both)
Purple dot (S1∧S2∧S3) — risk level using all 3 tests combined
Orange dashed vertical line — your chosen risk limit
Understanding the left-side sliders

Each of the 3 tests has two sliders:

▲ "Catches good upgrades" slider
How often this test correctly approves a genuine improvement. Set high (e.g. 90%) — you don't want good upgrades getting stuck. A small false rejection rate is acceptable.
▼ "Lets bad ones through" slider
How often this test accidentally passes a bad skill. This is the dangerous number — keep it low. Even 20% on a single test is fine when combined with two others.
What to try — step by step
1
Look at the dots on the chart. Notice how they march toward the bottom-left corner as you go from 1 → 2 → 3 tests.
2
Set "My Risk Limit" to 2%. The calculator tells you: how many tests do you need? Usually 2 is enough — because each test multiplies the risk reduction.
3
Drag the "Lets bad ones through" slider for Test 1 all the way to 50% (terrible test). Watch the teal dot shoot to the right — but the purple dot barely moves. The math: 50% × 15% × 12% = 0.9%.
4
Notice the bottom number cards — the "3 tests — risk" number is always far lower than the individual tests. That's the power of independence.
Why this matters for SAGE
The math works in your favour: If 3 tests each independently let 15% of bad upgrades through, the chance of all 3 failing simultaneously is 15% × 15% × 15% = 0.3% — 50× safer than using just one test. SAGE combines a rubric scorer, a holistic ranker, and an LLM judge for exactly this reason.

Pareto Frontier Explorer
How to choose between skills that trade accuracy against safety
The core idea

You're choosing a new laptop. You want maximum battery life AND maximum processing power. The problem: these goals compete. More powerful chips drain battery faster. Better battery means heavier components that slow things down. There is no laptop that wins on both.

So you face a real choice: what trade-off is right for you? A student might want battery. A video editor wants power.

SAGE faces the exact same problem. After training many versions of an AI skill, some score higher on accuracy (getting the right answer) but lower on safety (avoiding harmful outputs), and vice versa. Which version should you deploy?

The Pareto frontier is the set of options where you genuinely cannot do better on one measure without getting worse on the other. These are your only real choices — everything else is simply worse.

What you see on the chart
X →
Horizontal axis — Accuracy score (how often the skill gives the right answer). Farther right = more accurate.
Y ↑
Vertical axis — Safety score (how often outputs are appropriate and non-harmful). Higher up = safer.
Teal dots + teal curve — The Pareto frontier: the genuinely best options. Nothing beats any of these on BOTH measures simultaneously.
Dark dots — Dominated candidates: at least one teal option beats them on BOTH accuracy AND safety. You'd never rationally choose these.
Yellow dot — The skill you would deploy right now, given your current preference setting.
---
Dashed diagonal line — Your preference direction. It points toward what you value more.
Understanding the weight slider

The slider in the right panel asks: what matters more to you — accuracy or safety?

At 0% accuracy / 100% safety: you want the safest possible skill, even if it's less accurate. The yellow dot moves to the upper-left of the curve.

At 100% accuracy / 0% safety: you want the most accurate skill, even if it outputs riskier content. The yellow dot moves to the lower-right.

The deployed skill stats panel on the right shows the exact numbers for whichever point is currently selected.

What to try — step by step
1
Drag the weight slider slowly from left to right. Watch the yellow dot slide along the teal curve — you're literally moving your deployment decision in real time.
2
Notice the dark dots. No matter what your preference, you'd never pick one of these — a teal option is always objectively better.
3
Click "3D + Latency". Now bubble size = response speed. Bigger bubble = faster. Some accurate options are also slow — that changes the decision.
4
Click "Regenerate Skills" a few times. The shape of the curve changes — some populations are more spread out, some tightly clustered — but the frontier pattern is always there.
Why this matters for SAGE
The key insight: you don't need to retrain anything to change your deployment decision. The frontier is computed once. Your preference slider selects the right point on it. A medical product picks the upper-left (safety-first). An internal analytics tool might pick the lower-right (accuracy-first). Same trained skills, different deployment policies.

Evolution Generation Replay
How a population of AI skills gets better through mutation and selection — like biological evolution, but for prompts
The core idea

In biology, species evolve over generations. Random mutations occasionally produce an animal that's slightly better at surviving — it reproduces more, and the useful mutation spreads. Harmful mutations cause the animal to die before reproducing — they disappear. Over millions of generations, you get extraordinary complexity from a simple rule: keep what works, discard what doesn't.

SAGE applies exactly this logic to AI skill text (the instructions that tell the AI how to behave). Instead of animals, we have skill versions. Instead of survival, we have a quality gate. Each "generation" is one improvement cycle.

This replay shows 25 generations of that process with 40 skill versions competing simultaneously, so you can watch the population evolve in real time.

Reading the top chart (how quality improves)
Green line — The single best-performing skill in the population this generation
Blue line — Average quality across the whole population of skills
Shaded band — How spread out the population is (narrow = converged, wide = still exploring)
Red dashed line — Survival threshold (gate). Skills that mutate below this are rejected — their parent survives instead
X →
Each step = one generation (one improvement cycle)
Y ↑
Quality score 0.0 → 1.0 (higher = better)
Reading the bottom chart (where the population clusters)

Each vertical column in the bottom chart represents one generation. The brightness at each height shows how many skills are at that quality level. Brighter = more skills clustered there.

Watch the bright band shift upward over generations — that's the whole population getting better together, not just the top individual.

What each slider does
How many skills compete
More skills = more diversity = better chance of finding a great mutation, but slower to show clear patterns. Try 10 vs 80.
How big random changes are (mutation rate)
High = big jumps, explores widely but unstable. Low = tiny steps, converges slowly but reliably. Try 0.05 vs 0.45 and compare the shaded band.
Best skills kept unchanged (elitism)
A % of the top performers survive each generation without mutation. Prevents losing great discoveries. Set to 0% and watch the green line fluctuate more.
Minimum quality to survive (gate threshold)
Mutations below this score are rejected — the parent skill is kept instead. High gate = strict selection, slower early progress but very clean final population. Low gate = fast early gains, messier distribution.
What to try — step by step
1
Click Run with default settings. Watch the green line rise and the blue line follow. The shaded band should narrow over time — the population is converging.
2
Click Reset. Set mutation rate to max. Run again. The lines are noisier — big random jumps make it hard to converge. The band stays wide.
3
Reset again. Set elitism to 0%. Run. Notice the green "best" line can actually drop some generations — with no protected elites, great discoveries can be lost.
4
Reset. Set gate threshold to 0.70 (very strict). The population can't score below 0.70 to survive. Early progress is slow — but by generation 20 the whole population is clustered near the top in the bottom chart.
